Как relay работает

iPhone из России не может подключиться напрямую к DE/NL (заблокированы РКН). SPB relay — это nginx в режиме stream, который прозрачно проксирует TCP:

stream {
    upstream chameleon_de_tcp {
        server 162.19.242.30:2096;   # DE Xray VLESS
    }
    upstream chameleon_nl_tcp {
        server 194.135.38.90:2096;   # NL Xray VLESS
    }
    server {
        listen 2096;                  # iPhone → SPB:2096 → DE:2096
        proxy_pass chameleon_de_tcp;
    }
    server {
        listen 2098;                  # iPhone → SPB:2098 → NL:2096
        proxy_pass chameleon_nl_tcp;
    }
}
Ограничения relay: Только TCP. Не поддерживает UDP (Hysteria2, QUIC) и IPv6. Поэтому в конфиге клиента блокируется QUIC и отключён IPv6.

3. Путь пакета: от iPhone до сайта

1. iOS App хочет открыть google.com
   ↓
2. DNS запрос → 172.19.0.2:53 (TUN DNS)
   ↓
3. sing-box Router применяет правила:
   Rule 1: sniff → определяет протокол (DNS/TLS)
   Rule 2: protocol=dns → hijack-dns → перехват DNS
   ↓
4. sing-box DNS модуль
   → FakeIP возвращает 198.18.x.x мгновенно
   → Реальный DNS через dns-remote (1.1.1.1) через Proxy (VLESS)
   ↓
5. iOS App подключается к 198.18.x.x:443 (фейковый IP google.com)
   ↓
6. sing-box Router снова:
   Rule 1: sniff → TLS, domain: google.com
   Rule 2: не DNS → пропуск
   Rule 3: udp:443 → reject (если QUIC — блокируем)
   Rule 4: не приватный IP → пропуск
   Default: → Proxy (selector → Auto urltest)
   ↓
7. VLESS outbound (лучший по urltest, напр. "Russia → DE")
   → TCP соединение к 185.218.0.43:2096 (SPB relay)
   → Reality TLS handshake (SNI: ads.x5.ru, Vision flow)
   ↓
8. SPB Relay (nginx stream)
   → Прозрачный проброс TCP к 162.19.242.30:2096 (DE Xray)
   ↓
9. Xray на DE
   → Принимает VLESS Reality
   → Расшифровывает трафик
   → Отправляет запрос к google.com
   ↓
10. google.com отвечает → обратный путь

4. sing-box конфиг (что генерирует backend)

Backend (singbox.rs::generate_config()) генерирует JSON конфиг для iOS клиента:

4.1 Inbound (TUN)

{
  "type": "tun",
  "tag": "tun-in",
  "address": ["172.19.0.1/30"],        // IPv4 only! Без IPv6.
  "auto_route": true,
  "stack": "system",
  "mtu": 1400
}

4.2 DNS

{
  "servers": [
    {"tag": "dns-fake", "address": "fakeip"},
    {"tag": "dns-remote", "address": "https://1.1.1.1/dns-query",
     "address_strategy": "ipv4_only", "detour": "Proxy"},
    {"tag": "dns-direct", "address": "8.8.8.8",
     "address_strategy": "ipv4_only", "detour": "direct"}
  ],
  "rules": [
    {"outbound": "any", "server": "dns-fake"},
    {"outbound": "direct", "server": "dns-direct"}
  ],
  "fakeip": {
    "enabled": true,
    "inet4_range": "198.18.0.0/15"     // Без inet6_range!
  },
  "final": "dns-remote",
  "independent_cache": true
}
Почему FakeIP: FakeIP мгновенно возвращает фейковый IP (198.18.x.x) для любого домена. Реальный DNS-запрос идёт параллельно через VPN туннель (dns-remote → 1.1.1.1 через Proxy). Это убирает задержку на DNS resolution при открытии сайтов.

4.3 Route Rules (порядок критичен!)

{
  "default_domain_resolver": {"server": "dns-direct", "strategy": "ipv4_only"},
  "rules": [
    {"action": "sniff"},                                    // 1. Определить протокол
    {"protocol": "dns", "action": "hijack-dns"},            // 2. DNS → встроенный модуль
    {"network": "udp", "port": 443, "action": "reject"},    // 3. Блок QUIC
    {"ip_is_private": true, "outbound": "direct"}             // 4. Локальный трафик
  ]
}
Порядок правил критичен!
- hijack-dns ДОЛЖЕН быть ДО ip_is_private (иначе DNS к 172.19.0.2 уйдёт в direct и умрёт)
- QUIC reject ДОЛЖЕН быть ДО default route (иначе UDP 443 пойдёт через VLESS TCP и зависнет)

4.4 Outbounds

1. selector "Proxy"     — пользователь выбирает сервер (или Auto)
2. urltest "Auto"       — автовыбор лучшего по пингу (gstatic.com/generate_204)
3. vless "Russia → DE"  — 185.218.0.43:2096 (relay → DE)
4. vless "Russia → NL"  — 185.218.0.43:2098 (relay → NL)
5. vless "Germany"      — 162.19.242.30:2096 (direct, заблокирован из РФ)
6. vless "Netherlands"  — 194.135.38.90:2096 (direct, заблокирован из РФ)
7. hysteria2 "DE HY2"  — 162.19.242.30:8443 (UDP, timeout)
8. hysteria2 "NL HY2"  — 194.135.38.90:8443 (UDP, timeout)
9. direct               — прямое соединение (для приватных IP)

Каждый VLESS outbound содержит:

{
  "type": "vless",
  "server": "185.218.0.43",           // SPB relay IP
  "server_port": 2096,
  "uuid": "...",                       // UUID пользователя из БД
  "flow": "xtls-rprx-vision",         // Vision flow (без multiplex!)
  "tls": {
    "enabled": true,
    "server_name": "ads.x5.ru",       // SNI для Reality
    "utls": {"enabled": true, "fingerprint": "chrome"},
    "reality": {
      "enabled": true,
      "public_key": "...",             // X25519 public key (совпадает с Xray private key)
      "short_id": "..."               // Short ID пользователя из БД
    }
  }
}

5. iOS приложение — ключевые файлы

ФайлЧто делает
AppState.swiftГлавный стейт. initialize() → repairConfig → autoRegister → fetchConfig → toggleVPN
ConfigStore.swiftХранение конфига в App Group. saveConfig() → sanitize → write file + UserDefaults
APIClient.swiftHTTP к backend. registerDevice(), fetchConfig(), signInWithApple()
VPNManager.swiftNetworkExtension: load/connect/disconnect NETunnelProviderManager
ConfigSanitizer.swiftУбирает deprecated поля (sniff, domain_strategy, clash_api, auto_detect_interface)

ExtensionPlatformInterface.swiftМост libbox ↔ iOS. buildTunnelSettings(), openTun(), pathUpdate()

Поток подключения VPN

1. User taps "Connect"
   → AppState.toggleVPN()
   → VPNManager.connect(configJSON)
   ↓
2. iOS creates PacketTunnel process
   → ExtensionProvider.startTunnel(options)
   → ConfigSanitizer.sanitizeForIOS(config)
   ↓
3. ExtensionProvider.startSingBox(config)
   → LibboxSetup(basePath, workingPath, tempPath, logFile)
   → CommandServer.start()
   → startOrReloadService(config)    // blocks until started
   ↓
4. sing-box creates TUN (utun11)
   → ExtensionPlatformInterface.openTun()
   → buildTunnelSettings() — IPv4 only, DNS 172.19.0.2
   → setTunnelNetworkSettings()
   ↓
5. urltest pings all servers (gstatic.com/generate_204)
   → Best server selected (e.g. "Russia → NL" 157ms)
   ↓
6. NEVPNStatus → Connected
   → AppState.handleStatus() → commandClient.connect()
   → Background: silentConfigUpdate() (обновляет конфиг для след. раза)

9. Частые проблемы и решения

VPN "Connected" но сайты не грузят

  • Проверить DNS: в логах должно быть hijack-dns. Если нет — правило отсутствует или использован underscore.
  • Проверить reality verification failed — версия Xray несовместима с libbox.
  • Проверить IPv6/QUIC таймауты — должны быть заблокированы в route rules.

App не подключается вообще

  • Конфиг {"error":"User not found"} — юзер не в БД. Переустановить приложение.
  • unknown outbound type "dns" — dns outbound в конфиге. Удалить, использовать hijack-dns.

Xray в restart loop

  • unknown transport protocol: xhttp — Xray слишком старый (1.8.x). Нужен 25.x+.
  • Проверить docker logs xray для конкретной ошибки.

БД пустая после перезапуска

  • Проверить docker volume ls | grep pg — volume должен быть backend_pgdata.
  • Если был docker compose down -v — данные потеряны. Юзеры перерегистрируются автоматически.
