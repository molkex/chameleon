# Incident: Apple rejected v1.0.27 build 89 under Guideline 2.3

## Symptom

Apple Resolution Center message:

> The UIRequiredDeviceCapabilities key in the Info.plist is set up in such a way that the app will not install on the device used in review.
>
> Device type: iPhone 17 Pro Max, iPad Air 11-inch (M3)
> OS version: iOS 26.5

`appStoreState` went `WAITING_FOR_REVIEW` → `DEVELOPER_REJECTED`.

## Root cause

The source `MadFrogVPN/Info.plist` never declared `UIRequiredDeviceCapabilities`. Xcode 26 SDK auto-derived `[arm64]` at build time. The value was correct (both review devices are arm64), but Apple's automated review tooling apparently flags the **implicit** form for this specific check.

Verified locally by building Release with current sources, then `plutil`-inspecting the built `.app/Info.plist`:

```
UIRequiredDeviceCapabilities = (arm64)
UIDeviceFamily = (1)
MinimumOSVersion = 17.0
```

Same value, just not visible in the source plist Apple's tooling apparently inspects.

## Fix

1. Added explicit declaration to `MadFrogVPN/Info.plist`:
   ```xml
   <key>UIRequiredDeviceCapabilities</key>
   <array>
       <string>arm64</string>
   </array>
   ```
2. Bumped `CURRENT_PROJECT_VERSION` 89 → 90 in `project.yml`.
3. Archived + uploaded build 90 via [`../playbooks/ios-cli-release.md`](../playbooks/ios-cli-release.md).
4. Recovered submission via [`../playbooks/apple-reject-recovery.md`](../playbooks/apple-reject-recovery.md):
   - Cancelled rejected submission `7f541058-...` (PATCH `canceled=true`).
   - Attached build 90 to v1.0.27 (PATCH `relationships/build`).
   - Added v1.0.27 to existing empty draft submission `c32658c5-...` (POST `reviewSubmissionItems`).
   - PATCH `submitted=true`.
5. State went `WAITING_FOR_REVIEW` at 2026-05-28T07:49:10Z.
6. Apple approved and auto-released later same day. v1.0.27 build 90 now LIVE on App Store.

## Side-effects to clean up

- **IAPs stalled**: 4 non-renewing subscription products were first-submitted 2026-05-27 bundled with build 89. When build 89 was rejected, IAPs silently stayed in `WAITING_FOR_REVIEW` without progressing. Re-submitted standalone via `POST /v1/inAppPurchaseSubmissions` 2026-05-28 after build 90 approved.
- **`docs/TROUBLESHOOTING.md` top section** (added during this incident) warned about App Privacy disclosure being mandatory before next iOS build. **It was overcautious** — verified via Chrome MCP that Product Interaction + Other Diagnostic Data + Purchase History were already declared in ASC. EventTracker events fall under existing declarations. No action was needed for the privacy disclosure; the warning has been preserved as historical context.

## Lessons

- **Source plist values can differ from built plist values.** Always `plutil` the built `.app/Info.plist` if Apple's reject mentions any specific Info.plist key. The build process injects things at archive time.
- **Apple's automated review tooling is finicky about implicit defaults** even when the resolved value is correct. Declaring values explicitly in source removes any ambiguity at zero cost.
- **First-time IAPs are bundled with first app-version review.** When the app version is rejected, IAPs silently stall — they don't automatically re-process when the next version gets approved. Resubmit IAPs standalone (`POST /v1/inAppPurchaseSubmissions`) after the parent app is APPROVED.
- **Resolution Center reply is read-only after cancelling the submission.** Not a blocker — the new submission is a fresh request; reviewers see the latest binary regardless.

## Don't retry

- Don't try to DELETE a `reviewSubmissionItem` whose `state: REJECTED` — HTTP 409 "Item was already submitted". Cancel the parent submission instead.
- Don't try to POST a new `reviewSubmissionItem` referencing a version that's still attached to an existing (even rejected) submission — HTTP 409 "Item is already present in another reviewSubmission". Cancel the old submission first.
