## Pre-drill: bugs found and fixed before running anything live

Reading `failover.sh` closely (prompted by external advice from a second
model) surfaced 3 real bugs that would have broken the drill or masked a
real failure:

1. **`nl` target's health check hit nginx's port, but step 3 never started
   nginx** â€” only `docker start $CHAM` ran; NL's `chameleon-nginx` had been
   stopped since 06-29. Would have hung step 3 waiting on a health check
   nginx never served. Fixed: added a per-node `NGINX` var, started
   alongside chameleon when set.
2. **The "is current primary unknown" guard used POSIX `[ ]`, which doesn't
   glob-match** â€” `[ "$CUR" != unknown* ]` only matched the literal string
   `"unknown*"`, so an unreadable-MSK state fell through to `node_cfg` and
   died with a confusing "unknown node" error instead of a clear "can't
   read MSK, skipping fence" message. Fixed with `[[ ]]` (bash pattern
   match).
3. **The fence step's docker-stop command was one un-separated string** â€”
   `docker stop $OLD_CHAM docker stop chameleon-nginx` parsed as a single
   `docker stop` call with 4 arguments, silently erroring (harmlessly) on
   the literal `docker` and `stop`. Worked by accident. Fixed with an
   explicit `;`.

Also: NL's stopped `chameleon` container was running a **pre-2026-06-29
binary**, missing everything shipped since â€” including `655aa22` (the
refresh-reuse re-issue fix for the field logout loop). Promoting NL as-is
would have reintroduced that bug. Refreshed via the same lean
`Dockerfile.prebuilt` pattern used for WAW (cross-compiled locally, shipped,
rebuilt server-side, recreated with `docker create` â€” never started, so it
never touched the read-only replica DB). Container secrets were extracted
from the *existing* container's own env server-side and never left the box.

All 5 `infrastructure/failover/*.sh` scripts: `bash -n` + `shellcheck -S
warning` clean. Added `.github/workflows/infra-failover-shellcheck.yml` so
this doesn't regress silently. Closes roadmap `INFRA-FAILOVER-SHELLCHECK`.

## Also done before the drill: NL's dormant VPN exit (nl2) reactivated live

Per the same external advice: a dormant "reactivate on demand" fallback rots
silently (exactly what happened to `nl2`'s stale roster before the 05-31 GRA
incident of the same shape), while a *live* exit is continuously self-tested
by real client `urltest` traffic â€” the mechanism that actually kept VPN
service up through the 06-29 incident. Since NL's box is paid for regardless
(the `ss-ws` production router pins it there), a live nl2 costs nothing
extra.

Found and fixed the same class of bug GRA hit in
[2026-05-31-gra-france-user-sync.md](2026-05-31-gra-france-user-sync.md):
- `vpn_servers.nl2.user_api_url` was empty â€” set to `http://147.45.252.234:15380`.
- sing-box's `user-api` service was bound `127.0.0.1` only â€” rebound to
  `0.0.0.0`, config validated with `sing-box check` *before* restarting,
  restarted only the `singbox` container (the separate, 452-connection
  `singbox-ss-ws` router container was never touched).
- ufw opened `:15380` to WAW only (the `RelayUserSyncer` runs on WAW now,
  not NL â€” the old rule direction, if any, was stale).
- `is_active` flipped `true` in `vpn_servers`.
- WAW's `config.production.yaml` `relay.secrets` map was missing an `nl2`
  entry entirely (deployed post-06-29 without it) â€” sync was silently
  skipped every 30s, logged as `"no secret configured for remote exit"`.
  Added, following the existing `gra1`/`waw1` pattern (shared
  `USER_API_SECRET`). Restarted `chameleon-failover` to load it â€” brief
  blip, verified `/health` 200 before and after.

**Verified live, not just configured:** NL sing-box logs
`"users replaced on inbound/vless-reality-tcp: 10 users"` every ~30s
(matches the live active-subscriber count), and â€” found while checking â€”
**real production traffic was already flowing** through the `nl-via-msk`
relay chain to the reactivated exit (`device_04a0151d`, ~70ms to
`www.gstatic.com`) within minutes of turning it on.

`failover.sh`'s `EXITS` array updated to include NL (`147.45.252.234`) â€” it
was missing since 06-29, so a future failover wouldn't have repointed its
user-api ufw to a new primary.

## Post-drill: the failback silently broke the web frontend for ~1 hour

Caught only because the user forwarded a screenshot of the ops-alerts
Telegram channel (`@MadFrogRobot` â€” the same bot investigated in the cost
audit; its alert delivery is independent of the bot's own liveness, exactly
as documented there). The channel showed:

```
đŸ”´ WAW origin unreachable via Cloudflare (GRA vantage) â€” madfrog.online down.
   origin_health(madfrog.online via CFâ†’WAW)=521      [14:55 and 15:27 local]
```

**Root cause: my own bug fix from earlier today, applied incompletely.**
The fence step (bug #3 above) now correctly stops `chameleon-nginx` on
whichever node is the OLD primary. But `node_cfg`'s `waw` entry had
`NGINX=""` â€” inherited from a stale comment ("WAW serves chameleon directly
on :8000, no nginx yet") that predates 2026-07-01, when `chameleon-nginx`
became WAW's real web frontend for `madfrog.online` via Cloudflare. So
`failover.sh waw` (the failback) correctly restarted `chameleon-failover`
and passed its own health check (`:8000` directly, not through nginx) â€” but
never restarted `chameleon-nginx`, silently leaving the *web* frontend down
while the *API*/VPN path looked completely healthy.

Timing lines up almost exactly with the drill itself (failback ran
~11:50 UTC â‰ˆ 14:50 local) â€” this had been broken since the moment the drill
finished, for roughly an hour, until the screenshot prompted a check.
**My own post-drill verification only checked `api.madfrog.online/health`
and the mobile healthcheck endpoint â€” both bypass nginx entirely (MSK
routes straight to `:8000`) â€” so I never actually exercised the path that
broke.** That's the real lesson, not just the code bug.

Fixed both layers:
1. **Immediate:** `docker start chameleon-nginx` on WAW â€” verified
   `madfrog.online`, `/admin/app/`, `www.madfrog.online`, `mdfrog.site`,
   `grafana.madfrog.online` all correct again within minutes.
2. **Root cause:** `node_cfg`'s `waw` entry now sets
   `NGINX="chameleon-nginx"` too, so any future promotion/failback to WAW
   starts it automatically. `bash -n` + `shellcheck -S warning` clean.

**Process lesson, not just code:** any future failover/drill verification
MUST check `https://madfrog.online/` (the Cloudflareâ†’WAW web path) in
addition to `api.madfrog.online/health` (the MSKâ†’WAW API path) â€” they are
genuinely independent routes through genuinely independent WAW services,
and a passing API health check proves nothing about the web frontend.
