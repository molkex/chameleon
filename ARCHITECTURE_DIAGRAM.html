3. DNS Flow — The Bug and Fix

BEFORE (BROKEN)

1
App wants to open google.com
2
DNS query → 172.19.0.2:53 (TUN DNS)
3
Route rule 1: sniff → detected as DNS
4
Rule 2: ip_is_private=true → direct
172.19.0.2 is private IP → matches!
5
DNS goes to "direct" outbound
Direct sends to real 172.19.0.2 → NOTHING THERE
6
DNS resolution fails
No domain resolves → no sites load

Route rules: sniff → ip_is_private → direct

AFTER (FIXED)

1
App wants to open google.com
2
DNS query → 172.19.0.2:53 (TUN DNS)
3
Route rule 1: sniff → detected as DNS
4
Rule 2: protocol=dns → action: hijack-dns
DNS intercepted BEFORE ip_is_private! No dns outbound needed.
5
hijack-dns → sing-box DNS module
FakeIP returns 198.18.x.x instantly
6
Real DNS via Proxy
dns-remote resolves via 1.1.1.1 through VPN

Route rules: sniff → dns→hijack-dns → ip_is_private → direct

ROOT CAUSE: DNS queries to 172.19.0.2 (TUN interface) were caught by ip_is_private rule and sent to direct outbound where they died. VPN showed "Connected" but no sites loaded.

FIX (2026-04-06): Added route rule {"protocol": "dns", "action": "hijack-dns"} BEFORE ip_is_private. The hijack-dns action (sing-box 1.11+) intercepts DNS and routes it to the built-in DNS module. No dns outbound needed — it was removed in sing-box 1.13. DNS now goes to FakeIP + remote resolution through Proxy.
